# 内容质量标准

## 写作规范

### Markdown 格式规范

#### 标题层级
```markdown
# H1: 课程标题（每个文档只用一次）
## H2: 模块标题
### H3: 知识点标题
#### H4: 子主题
##### H5: 细节（尽量不用）
###### H6: 极少使用
```

**原则**:
- 不跳级（H1 → H3 ❌）
- 不少于3级（H1 → H2 → H4 ❌）
- 保持层次清晰

#### 文本格式

**加粗**：
```markdown
**重要概念**
**关键术语**
**注意事项**
```

*斜体*：
```markdown
*强调内容*
_英文术语_
```

`代码`：
```markdown
`代码片段`
`配置项`
`命令`
```

#### 列表规范

**无序列表**：
```markdown
- 要点1
- 要点2
  - 子要点2.1
  - 子要点2.2
```

**有序列表**：
```markdown
1. 步骤1
2. 步骤2
3. 步骤3
```

**任务列表**：
```markdown
- [ ] 未完成任务
- [x] 已完成任务
```

#### 代码块规范

**带语言标注**：
````markdown
```java
public class Example {
    // Java 代码
}
```

```python
def example():
    # Python 代码
    pass
```

```bash
bash command-here
```

```yaml
key: value
```
````

**必须包含的语言标签**:
- Java: `java`
- Python: `python`
- Bash: `bash`
- YAML: `yaml`
- JSON: `json`
- Markdown: `markdown`
- SQL: `sql`
- XML: `xml`
- Properties: `properties`

#### 表格规范

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 内容1 | 内容2 | 内容3 |
| 内容4 | 内容5 | 内容6 |
```

**原则**:
- 对齐工整
- 表头明确
- 不使用空单元格（用 `-` 或 `N/A`）

#### 引用规范

```markdown
> 普通引用
>
> 多行引用

> **嵌套引用**
>> 二级嵌套
```

#### 链接规范

**内部链接**：
```markdown
[链接文本](#anchor-id)
[知识点1.1](#知识点11-核心概念)
```

**外部链接**：
```markdown
[链接文本](https://example.com)
[Spring官方文档](https://spring.io/projects/spring-boot)
```

**参考式链接**（推荐用于频繁使用）：
```markdown
[Spring官方文档][1]
[GitHub][2]

[1]: https://spring.io/projects/spring-boot
[2]: https://github.com
```

## 内容组织原则

### 结构完整性

**每个知识点必须包含**:
1. ✅ 学习目标（3-5条）
2. ✅ 核心内容（定义 + 原理 + 实践）
3. ✅ FAQ（4-5个）
4. ✅ 思考题（3-5个）
5. ✅ 扩展阅读（可选）

### 内容逻辑

**从易到难**:
```
概念 → 原理 → 实践 → 优化 → 扩展
```

**从理论到实践**:
```
定义 → 工作原理 → 源码分析 → 实战示例
```

### 可读性

**段落长度**:
- 每段 3-5 句话
- 段落之间空一行
- 避免大段文字

**句子长度**:
- 每句不超过 30 字
- 简单句为主
- 避免嵌套从句

**过渡自然**:
- 使用过渡词："因此"、"然而"、"此外"
- 使用承上启下句
- 保持逻辑连贯

## 代码质量标准

### 代码示例要求

#### 完整性
✅ **必须包含**:
- 完整的可运行代码
- 必要的 import 语句
- 包名和类名
- main 方法（如需要）

❌ **避免**:
- 代码片段（无法运行）
- 省略关键部分
- "..." 省略号

#### 注释规范

```java
/**
 * 类描述
 * 说明这个类的作用、使用场景
 */
public class Example {

    /**
     * 方法描述
     * @param param1 参数说明
     * @return 返回值说明
     */
    public Result method(String param1) {
        // 关键步骤注释
        Step1();  // 说明步骤1的作用

        Step2();  // 说明步骤2的作用

        return result;
    }
}
```

**注释原则**:
- 注释 "为什么"，不是 "是什么"
- 关键逻辑必须注释
- 复杂算法必须注释
- 不注释显而易见的代码

#### 代码风格

**Java**:
- 遵循 Google Java Style
- 4 空格缩进
- 大括号不换行
- 变量命名 camelCase

**Python**:
- 遵循 PEP 8
- 4 空格缩进
- 变量命名 snake_case
- 文档字符串

**Bash**:
- 4 空格缩进
- 变量大写 + 下划线
- 函数命名 snake_case

## FAQ 编写规范

### FAQ 选择标准

**好的 FAQ**:
- ✅ 初学者常问的问题
- ✅ 容易混淆的概念
- ✅ 常见错误和解决方案
- ✅ 实践中的坑

**避免的 FAQ**:
- ❌ 太过简单（显而易见）
- ❌ 太过偏门（极少遇到）
- ❌ 重复知识点的已有内容

### FAQ 格式

```markdown
### FAQ

**Q1: [问题标题]?**

**A**: [详细解答]

**说明**:
- 问题原因
- 解决方案
- 示例代码
- 最佳实践

**相关链接**:
- [相关知识点](#链接)
- [参考资料](#链接)

---

**Q2: [问题标题]?**

**A**: [详细解答]
...
```

### FAQ 数量

- 每个知识点 4-5 个
- 整个模块 20-30 个
- 整门课程 60-100 个

## 思考题设计

### 思考题类型

#### 理解性问题
- 检查概念理解
- 比较不同方案
- 分析优缺点

示例：
1. A 和 B 有什么区别？
2. 为什么选择方案A而不是B？
3. 这种设计有什么优缺点？

#### 实践性问题
- 应用所学知识
- 解决实际问题
- 设计解决方案

示例：
1. 如何实现 [功能]？
2. 如果遇到 [问题]，如何排查？
3. 如何优化 [性能]？

#### 扩展性问题
- 深入研究
- 探索前沿
- 创新思考

示例：
1. 这个技术未来会如何发展？
2. 是否有其他替代方案？
3. 如何改进现有设计？

### 思考题数量

- 每个知识点 3-5 个
- 难度递进
- 理论 + 实践 + 扩展 各占1/3

## 源码分析规范

### 源码选择

**选择标准**:
- ✅ 核心类和方法
- ✅ 典型设计模式
- ✅ 重要算法
- ✅ 常用API

**避免**:
- ❌ 过于简单（getter/setter）
- ❌ 过于复杂（>100行）
- ❌ 不常用功能

### 源码分析格式

```markdown
#### 源码分析: [类/方法名]

**所在位置**:
\`\`\`
org.springframework.boot.SpringApplication
\`\`\`

**核心代码**:
\`\`\`java
public ConfigurableApplicationContext run(String... args) {
    // 1. 启动计时器
    StopWatch stopWatch = new StopWatch();
    stopWatch.start();

    // 2. 创建 BootstrapContext
    // ... 代码注释

    // 3. 准备 Environment
    // ... 代码注释

    return context;
}
\`\`\`

**逐行解析**:

**1. 启动计时器**（第1-2行）
- 目的：统计启动时间
- 实现：使用 Spring 的 StopWatch
- 输出：启动耗时日志

**2. 创建 BootstrapContext**（第3-5行）
- 目的：启动前的准备
- 实现：使用 Bootstrapper 扩展点
- 说明：这是Spring Boot 2.4+的新特性

**设计亮点**:
1. ✅ 模板方法模式：run方法定义流程
2. ✅ 扩展点设计：Bootstrap、Initializer、Listener
3. ✅ 异常处理：统一处理启动异常

**最佳实践**:
- 使用 StopWatch 统计耗时
- 分阶段初始化（降低复杂度）
- 提供扩展点（保持灵活性）
```

### 注释详细度

**适度注释原则**:
- 关键逻辑：✅ 详细注释
- 简单逻辑：⚠️ 简要注释
- 显而易见：❌ 不注释

**注释内容**:
- 为什么这样做（设计意图）
- 什么场景使用（使用场景）
- 有什么注意点（坑和陷阱）

## 最佳实践章节

### 内容要求

每个知识点或模块应该包含：

#### 推荐做法 ✅
```markdown
#### ✅ 推荐做法

1. **命名规范**
   ```java
   // Good
   UserService userService;
   ```

2. **异常处理**
   ```java
   // Good
   try {
       // code
   } catch (SpecificException e) {
       // handle
   }
   ```

3. **资源管理**
   ```java
   // Good
   try (Resource r = new Resource()) {
       // use r
   }  // auto close
   ```
```

#### 不推荐做法 ❌
```markdown
#### ❌ 不推荐做法

1. **避免魔法值**
   ```java
   // Bad
   if (status == 1) { }

   // Good
   if (status == Status.ACTIVE) { }
   ```

2. **避免吞异常**
   ```java
   // Bad
   try { } catch (Exception e) { }

   // Good
   try { } catch (Exception e) {
       logger.error("msg", e);
   }
   ```
```
